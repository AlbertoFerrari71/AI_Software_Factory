# 01 — Vision

## 1. Visione

AI Software Factory vuole diventare un sistema operativo di sviluppo software assistito da AI.

Non deve limitarsi a "scrivere codice". Deve gestire l'intero ciclo:

```text
Idea → Chiarezza → Requisiti → Architettura → Task → Codice → Test → Documentazione → Rilascio
```

La visione è creare un metodo in cui l'intelligenza artificiale accelera il lavoro, ma ogni passaggio rimane:

- comprensibile;
- controllabile;
- tracciabile;
- testabile;
- reversibile.

---

## 2. Frase guida

> From Idea to Tested Code — with Human Control.

Traduzione operativa:

> Dall'idea al codice testato, senza perdere il controllo umano.

La bussola culturale sull'AI moderna, sulla comprensione funzionale e sull'AI come collaboratrice verificabile e' in `docs/project_context/VISIONE_OPERATIVA_AI.md`.

## 4. Perché "Factory"

Il termine Factory indica:

- processo ripetibile;
- standardizzazione;
- controllo qualità;
- output verificabile;
- miglioramento continuo;
- strumenti integrati;
- produzione non casuale.

---

## 5. Perché "Alchemy"

Il termine Alchemy indica la trasformazione:

```text
idea grezza → struttura → requisiti → architettura → task → codice affidabile
```

L'alchimia non è magia incontrollata. È una metafora del processo di raffinazione.

---

## 6. Principi fondamentali

### 6.1 Human-in-the-loop

L'AI può proporre, scrivere, controllare, riassumere e automatizzare.

L'umano mantiene il controllo su:

- obiettivi;
- architettura;
- sicurezza;
- merge;
- deploy;
- cancellazioni;
- dati reali;
- credenziali;
- decisioni di prodotto.

### 6.2 Small steps

Ogni lavoro deve essere piccolo, numerato e verificabile.

Esempio:

```text
010) Visione e contesto
020) Repository Genesis
030) Safety Model
040) Prompt Packet Generator
```

### 6.3 Safety by default

Le automazioni devono partire protette:

- read-only prima;
- dry-run quando possibile;
- branch dedicati;
- test obbligatori;
- approvazione esplicita per azioni rischiose;
- rollback definito.

### 6.4 Documentation as memory

La documentazione non è un'aggiunta finale. È la memoria del progetto.

Serve a:

- far riprendere il lavoro dopo giorni o settimane;
- istruire Codex;
- evitare errori ripetuti;
- tracciare decisioni;
- facilitare PR e review.

### 6.5 Reuse

Ogni progetto migliora il framework.

Il learning loop aggiorna:

- prompt;
- checklist;
- template;
- policy;
- test strategy;
- decision log.

---

## 7. Local-first, SaaS-ready

La prima versione deve essere locale/personale.

Motivi:

- meno complessità;
- meno costi;
- meno rischi;
- più controllo;
- feedback più rapido.

Ma deve essere progettata per evolvere in SaaS.

Elementi SaaS-ready da prevedere:

- workspace;
- utenti;
- ruoli;
- permessi;
- audit log;
- limiti consumo;
- billing;
- secret vault;
- connettori;
- template marketplace;
- sandbox esecuzione codice;
- integrazioni GitHub/GitLab.

Non vanno però implementati tutti nel primo MVP.

---

## 8. Criterio di successo a lungo termine

AI Software Factory sarà maturo quando un utente potrà:

1. descrivere un'idea;
2. rispondere a poche domande guidate;
3. ottenere un progetto strutturato;
4. generare issue e task;
5. far lavorare Codex in modo controllato;
6. verificare test e qualità;
7. aggiornare documentazione;
8. rilasciare una versione;
9. riusare il metodo su un altro progetto.

---

## 9. Anti-visione

Il progetto non deve diventare:

- una dashboard bella ma vuota;
- un framework astratto mai usato;
- un generatore di codice senza test;
- un sistema con troppi livelli prima del MVP;
- un SaaS prematuro;
- una dipendenza rigida da un singolo tool;
- un agente con eccessivi privilegi.
